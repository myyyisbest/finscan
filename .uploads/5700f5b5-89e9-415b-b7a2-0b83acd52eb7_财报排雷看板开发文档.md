# 财报排雷看板 — 开发文档

> **写给接手此项目的新人**：本文档是"财报排雷"（Finscan）功能的完整规格说明。目标是将现有的 Claude Code Skill 改造为一个独立的 Web 看板应用，内置完整的排雷分析能力。

---

## 1. 项目背景与核心理念

### 1.1 方法论来源

本项目的方法论源自唐朝老师《手把手教你读财报（新准则升级版）》。核心思想：

> **"财报是用来排除企业的，不是用来发现牛股的。"**

这不是一个"选股"工具，而是一个**排雷工具**。它的目标是从财务数据中识别欺诈信号和风险信号，帮助投资者排除有问题的公司。即使全部 PASS 也不等于"好公司"——只等于"未发现明显雷区"。

### 1.2 判定哲学：有疑就杀

规则体系中大量使用**信号叠加**来触发告警。单一指标的异常可能是行业特性或会计处理差异；但多个信号同时出现时，财务造假概率急剧上升。

举例：毛利率上升 + 应收账款增速超过营收增速 + 应付账款下降 → 三信号叠加 → 极高造假概率。

### 1.3 现有实现 vs 目标形态

| 维度 | 现有实现（Claude Code Skill） | 目标形态（Web 看板） |
|------|------------------------------|---------------------|
| 交互方式 | CLI `/finscan 600519` | Web UI，搜索框输入 |
| 执行环境 | LLM 驱动，逐步调用脚本 | 后端服务 + 定时任务 |
| 数据展示 | Markdown 文本报告 | 可视化图表 + 交互式看板 |
| 历史追踪 | 每次独立分析 | 数据库持久化，可对比多期 |
| 用户管理 | 无 | 登录 + 自选股列表 + 监控告警 |
| 可扩展性 | 依赖 Claude Code | 标准 REST API |

## 3. 数据源详解

### 3.1 数据源策略

采用 **结构化 API（主）+ 年报 PDF（辅）** 双数据源：

| 数据源 | 覆盖规则 | 优势 | 限制 |
|--------|---------|------|------|
| Tushare Pro API | Layer 0-4, 部分 5-6（约 23/30 规则） | 零提取误差、10 年历史、自动同行对比 | 需要 Token（免费注册，有调用频率限制） |
| akshare（东方财富/同花顺） | Layer 0-4, 部分 5-6 | 完全免费，无需 Token | 字段覆盖略少于 Tushare，稳定性稍差 |
| 年报 PDF（巨潮资讯网） | Rule 1.3, 5.3, 5.4, 5.5, 5.6, 5.8, 5.9, 6.2 | 非结构化定性数据 | 需要 PDF 下载 + 文本提取 + 正则解析 |

### 3.2 数据获取流程图

```
输入: 股票代码 (如 600519)
         │
         ├── 1. 尝试 Tushare API (如 TOKEN 存在)
         │      └── stock_basic → fina_audit → income → balancesheet
         │          → cashflow → fina_indicator → top10_holders → peers
         │
         ├── 2. Tushare 失败 → 降级到 akshare (免费)
         │      └── 同等字段输出
         │
         ├── 3. 下载年报 PDF (巨潮资讯网 cninfo.com.cn)
         │      └── 搜索 → 匹配年度 → 下载 → pdfplumber 提取文本
         │
         └── 4. 数据标准化 → 入库 → 触发规则引擎
```

## 4. 规则引擎：7 层 30 条规则详解

> **这是整个系统的核心**。每条规则都必须在代码中精确实现。以下逐条说明判定逻辑、阈值和所需数据。

---

### Layer 0 — 门槛检查（一票否决）

Layer 0 的 FAIL 意味着**无需继续分析**——公司的基本可信度已经瓦解。

#### Rule 0.1 — 审计意见

```
数据: data.audit[latest].audit_result
逻辑: 必须为"标准无保留意见"
PASS: 审计意见不含"保留"以外的修饰词
FAIL: 保留意见 / 无法表示意见 / 否定意见 → 直接排除
```

审计是财报可信度的底线。非标准意见 = 公司财务有重大问题，会计师不愿背书。

#### Rule 0.2 — 按时披露

```
数据: data.audit[latest].ann_date vs data.audit[latest].end_date + 4个月
逻辑: 年报必须在次年4月30日前披露
PASS: ann_date <= 次年0430
FAIL: 超期披露
```

迟交年报通常是问题暴露的前兆——公司需要更多时间"调整"数据，或审计师在博弈。

---

### Layer 1 — 利润表信号（6 条规则）

#### Rule 1.1 — 毛利率异常高 + 大幅波动

```
数据: indicators[].grossprofit_margin + peers.peers[].grossprofit_margin
计算:
  gm_current   = 最新年度毛利率
  gm_prev      = 上一年度毛利率
  gm_yoy       = gm_current - gm_prev  (百分点变化)
  peer_median  = 同行毛利率中位数
  gm_vs_peer   = gm_current - peer_median

PASS: |gm_yoy| ≤ 5pp 且 gm_vs_peer ≤ 15pp
WARN: |gm_yoy| > 5pp 或 gm_vs_peer > 15pp
FAIL: |gm_yoy| > 10pp 且 gm_vs_peer > 15pp
```

**背后逻辑**: 毛利率远超同行说明要么有真壁垒（如茅台），要么在财务造假。大幅波动说明壁垒不稳定或有人在操纵。

#### Rule 1.2 — 毛利率↑ + 应收↑ + 应付↓（三信号叠加）

```
数据: indicators[].grossprofit_margin + balance[].accounts_receiv + balance[].acct_payable
条件:
  A: 毛利率 YoY 上升
  B: 应收账款增速 > 营收增速
  C: 应付账款 YoY 减少

PASS: 满足 0-1 个条件
WARN: 满足 2 个条件
FAIL: 3 个条件同时满足 → 触发 +6 组合加分
```

**背后逻辑**: 这是经典的"虚增收入"信号链。毛利率上升却收不到现金（应收增长），还失去了对供应商的议价能力（应付下降），三者叠加 = 高度可疑。

#### Rule 1.3 — 运费增长 vs 收入增长

```
数据: PDF 年报附注（销售费用/营业成本的运费明细）
      注: Tushare 无此字段，必须从 PDF 提取
计算:
  freight_growth = 运费 YoY 增速
  revenue_growth = 营收 YoY 增速

PASS: freight_growth >= revenue_growth × 50%
WARN: freight_growth < revenue_growth × 50%
FAIL: freight_growth < revenue_growth × 25%
SKIP: 运费未在附注中单独披露
```

**背后逻辑**: 卖货需要运输。如果收入增长 50% 但运费只增长 5%，那多出来的收入是怎么送到客户手里的？这是制造业常用的造假检验。

**PDF 提取指引**: 在年报附注中搜索 `运输费` / `运费` / `装卸费`，通常出现在"销售费用"或"营业成本"的明细拆分中。

#### Rule 1.4 — 其他业务收入占比突增

```
数据: income[].oth_biz_income + income[].revenue
计算:
  ratio        = oth_biz_income / revenue × 100
  ratio_prev   = 上一年度同比
  ratio_change = ratio - ratio_prev

PASS: ratio ≤ 5% 或 ratio_change ≤ 3pp
WARN: ratio > 5% 且 ratio_change > 3pp
FAIL: ratio > 15% 或 ratio_change > 10pp
```

**背后逻辑**: 其他业务收入是"垃圾桶"科目。正常公司这个科目很小且稳定。突然飙升意味着公司在用非常规手段做收入。

#### Rule 1.5 — 费用率异常下降

```
数据: income[].sell_exp + admin_exp + fin_exp + revenue
计算:
  expense_ratio  = (sell_exp + admin_exp + fin_exp) / revenue × 100
  avg_3yr        = 近3年费用率均值
  drop           = avg_3yr - expense_ratio_current

PASS: drop ≤ 3pp
WARN: drop > 3pp 或费用率显著低于同行中位数
FAIL: drop > 5pp
```

**背后逻辑**: 费用率突然大幅下降而收入增长，说明公司可能在**费用资本化**（把当期费用伪装成资产），或者**少计费用**。正常经营中费用率通常是稳定的。

#### Rule 1.6 — 资产减值损失暴增

```
数据: income[].assets_impair_loss + credit_impa_loss + n_income_attr_p
计算:
  total_impair    = |assets_impair_loss| + |credit_impa_loss|
  impair_yoy      = YoY 变动百分比
  impair_to_profit = total_impair / |n_income_attr_p| × 100

PASS: impair_yoy ≤ 50% 且 impair_to_profit ≤ 5%
WARN: impair_yoy > 50%
FAIL: impair_yoy > 100% 或 impair_to_profit > 5%
```

**背后逻辑**: 减值是把以前的"水分"一次性挤掉。减值暴增 = 以前的资产质量有问题（或者在为以后的"轻装上阵"做准备——俗称"洗大澡"）。

---

### Layer 2 — 现金流量表信号（3 条规则）

#### Rule 2.1 — 经营 CF 优异 + 投资 CF 持续大额为负

```
数据: cashflow[].n_cashflow_act + n_cashflow_inv_act (5-10年)
计算:
  逐年: n_cashflow_act > 0 且 |n_cashflow_inv_act| > n_cashflow_act × 0.8
  统计满足条件的年数

PASS: 满足年数 ≤ 1
WARN: 满足年数 2-3
FAIL: 满足年数 ≥ 4
```

**背后逻辑**: 经营 CF 良好却持续大额投资支出，要警惕投资是否有效。如果多年投资都不产生回报，这些钱可能去了不该去的地方。

#### Rule 2.2 — 经营 CF 持续为负

```
数据: cashflow[].n_cashflow_act (5-10年)
计算:
  统计 n_cashflow_act < 0 的年数
  统计连续为负的最长年数

PASS: 近5年为负年数 ≤ 1
WARN: 近5年为负年数 = 2
FAIL: 连续 ≥ 3 年为负
```

**背后逻辑**: 经营 CF 是企业自我造血能力的最终体现。持续为负意味着企业无法靠日常经营产生现金，必须不断融资或卖资产续命。

#### Rule 2.3 — 高现金 + 高息借债（大存大贷）

```
数据: balance[latest].money_cap + st_borr + lt_borr + bond_payable
计算:
  interest_bearing_debt = st_borr + lt_borr + bond_payable
  cash   = money_cap
  implied_rate = |fin_exp| / interest_bearing_debt × 100

PASS: cash < interest_bearing_debt × 0.5 或 implied_rate ≤ 基准+2pp (基准取 ~3-4%)
WARN: cash > interest_bearing_debt × 0.5 且 implied_rate > 基准+2pp
FAIL: cash > interest_bearing_debt 且 implied_rate > 基准+4pp
  → FAIL 且 Rule 4.1 也 FAIL → 触发 +8 组合加分
```

**背后逻辑**: 这是 A 股最经典的造假信号。账上有大量现金却用高利率去借钱——如果现金是真的，为什么不还贷？如果贷款是真的，那现金去哪了？康得新、康美药业的经典案例。

---

### Layer 3 — 资产负债表信号（5 条规则）

#### Rule 3.1 — 应收增速 > 收入增速

```
数据: balance[].accounts_receiv + income[].revenue
计算:
  ar_growth  = 应收账款 YoY 增速
  rev_growth = 营收 YoY 增速

PASS: ar_growth / rev_growth ≤ 1.5 (当 rev_growth > 0) 或应收 < 营收 5%
WARN: ar_growth / rev_growth > 1.5
FAIL: ar_growth / rev_growth > 2.0
边界: 营收下降但应收增长 → 直接 WARN
```

**背后逻辑**: 正常的增长应收和营收大致同步。应收增速远超收入 = 公司在**放宽信用政策**冲收入（甚至虚构收入后挂在应收上）。

#### Rule 3.2 — 存货周转率↓ + 毛利率↑（造假黄金组合）

```
数据: indicators[].inv_turn + grossprofit_margin
计算:
  inv_turn_change = 存货周转率 YoY 变化%
  gm_change        = 毛利率 YoY 变化(pp)

PASS: 不满足组合条件
WARN: inv_turn_change < -10% 且 gm_change > 0
FAIL: inv_turn_change < -20% 且 gm_change > 3pp
  → FAIL 触发 +10 特殊加分（权重最高的单一组合）
```

**背后逻辑**: 正常经营中毛利率上升意味着产品好卖，存货应该周转更快。但毛利率升 + 周转率降 = 产品"好卖"却堆积在仓库——矛盾。最大的可能是**虚构收入**导致毛利率虚高、存货虚增（因为虚构的成本没地方去）。

#### Rule 3.3 — 在建工程不转固

```
数据: balance[].cip + fix_assets (3-5年)
计算:
  逐年: CIP 增长 > 30% 但固定资产未相应增长
  统计持续年数

PASS: CIP 不存在或未持续增长
WARN: CIP 增长 > 30% 且固定资产增长率 < CIP 增长率的 50%
FAIL: 上述情况持续 ≥ 3 年
```

**背后逻辑**: 在建工程是造假的"黑洞"科目。工程完工后应转入固定资产并开始计提折旧（减少利润）。如果一直不转固，就能避免折旧，同时掩盖工程可能根本不存在的事实。

#### Rule 3.4 — 长期待摊费用大增

```
数据: balance[].lt_amort_deferred_exp
计算:
  yoy_change = YoY 变动%

PASS: yoy_change ≤ 50% 或金额 < 总资产 1%
WARN: yoy_change > 50%
FAIL: yoy_change > 100%
```

**背后逻辑**: 长期待摊费用是把钱花出去了但不计入当期费用，而是分多年摊销。大增意味着公司可能把当期费用"藏"到这个科目里，粉饰利润。

#### Rule 3.5 — 坏账计提比例低于同行

```
数据: income[].credit_impa_loss + balance[].accounts_receiv + peers
计算:
  bad_debt_ratio = |credit_impa_loss| / accounts_receiv × 100

PASS: 不显著低于同行
WARN: 低于同行中位数
FAIL: 低于同行中位数 50%
SKIP: 应收账款极小或同行数据不足
```

**背后逻辑**: 坏账准备是利润的"蓄水池"。计提太少 → 利润虚高；计提太多 → 在为来年"储存"利润。明显低于同行意味着公司在少计提坏账以美化利润。

### Layer 4 — 交叉验证（5 条规则，权重最高）

Layer 4 是排雷体系中最关键的一层。它不单独看某一报表，而是**交叉验证三张报表之间的逻辑一致性**。

#### Rule 4.1 — 经营 CF / 净利润 < 1

```
数据: cashflow[].n_cashflow_act + income[].n_income_attr_p (5+年)
计算:
  逐年: cf_to_profit = n_cashflow_act / n_income_attr_p (净利润 > 0 时)
  统计 cf_to_profit < 1 的年数

PASS: 近5年仅 0-1 年 < 1
WARN: 近5年 2 年 < 1
FAIL: 连续 ≥ 3 年 < 1
```

**背后逻辑**: 这是**最核心的验证**。如果净利润是真的，它应该产生相应的现金。优秀企业此比值持续 > 1（经营 CF 多于净利润，因为有折旧等非现金费用）。持续 < 1 = 利润含金量低。

#### Rule 4.2 — 销售收现 / 营收 < 1

```
数据: cashflow[].c_recp_prov_sg_act + income[].revenue
计算:
  cash_to_rev = c_recp_prov_sg_act / revenue
  (简化版，不含增值税调整)

PASS: cash_to_rev ≥ 0.9 (持续多年)
WARN: cash_to_rev < 0.9
FAIL: cash_to_rev < 0.8 持续 ≥ 2 年
```

**背后逻辑**: 卖了东西应该收到钱。销售收现 ÷ 营收长期 < 1 意味着大量收入以应收账款形式存在，收不回现金。

#### Rule 4.3 — 利润膨胀 → 资产膨胀

```
数据: income[].revenue + n_income_attr_p + balance[].total_assets
计算:
  asset_growth  = 总资产 YoY 增速
  rev_growth    = 营收 YoY 增速
  profit_growth = 净利润 YoY 增速

PASS: asset_growth ≤ rev_growth × 1.5 或利润未增长
WARN: asset_growth > rev_growth × 2 且利润增长
FAIL: asset_growth > rev_growth × 3 且利润增长，连续 ≥ 2 年
```

**背后逻辑**: 虚增利润必然导致资产负债表某处膨胀（因为借贷必相等）。如果利润增长但资产增速远超收入增速，利润可能是"做"出来的。

#### Rule 4.4 — 核心利润 vs 净利润背离

```
数据: income[].revenue + oper_cost + sell_exp + admin_exp + fin_exp + n_income_attr_p
计算:
  core_profit = revenue - oper_cost - sell_exp - admin_exp - fin_exp
  divergence  = |core_profit - n_income_attr_p| / |n_income_attr_p| × 100

PASS: divergence ≤ 20%
WARN: divergence > 20%
FAIL: divergence > 40%
```

**背后逻辑**: 核心利润是主营业务赚的钱。大幅背离说明净利润主要来自投资收益、公允价值变动、营业外收入等非经常性项目——这些是不可持续的。

#### Rule 4.5 — 净利润增长 + FCF 持续为负

```
数据: income[].n_income_attr_p + indicators[].fcff
计算:
  逐年: 净利润 > 0 且增长，但 FCF < 0
  统计满足条件的年数

PASS: 满足 0-1 年
WARN: 满足 2 年
FAIL: 满足 ≥ 3 年
```

**背后逻辑**: FCF = 经营 CF - 资本支出。利润在增长但自由现金流持续为负 = 赚的钱全部（甚至更多）投回去了，而且看不到回报。

### Layer 5 — 非财务信号（9 条规则）

Layer 5 主要是定性信号，部分需要 PDF 年报文本。

#### Rule 5.1 — 更换审计机构

```
数据: audit[].audit_agency (多年)
逻辑: 比较各年度事务所名称

PASS: 多年同一家
WARN: 更换过 1 次
FAIL: 更换 ≥ 2 次
```

频繁换所 = 公司可能在"炒"不配合的审计师。尤其关注从"四大"换到小所。

#### Rule 5.2 — 大股东减持

```
数据: holders[] (最近 2-3 期)
逻辑: 比较第一大股东 hold_ratio 变化

PASS: 稳定或增加
WARN: 连续下降
FAIL: 大幅下降 (> 5pp)
```

最了解公司的人正在跑路。

#### Rule 5.3 — 财务总监频繁更换

```
数据: PDF 年报 "董事、监事、高级管理人员" 章节
搜索: "财务总监" / "首席财务官" / "CFO"
逻辑: 查看是否标注"本年离任"

PASS: 无异常变动
WARN: 本年变更
FAIL: 近 2-3 年多次变更
SKIP: 无法从当期年报判断
```

#### Rule 5.4 — 独董集体辞职

```
数据: PDF 年报 "董事、监事、高级管理人员" 章节
搜索: "独立董事" → 是否多人标注"辞职"

PASS: 无异常
WARN: 1 名独董辞职
FAIL: ≥ 2 名独董同期辞职
SKIP: 无法判断
```

独董辞职是重大风险信号——他们不愿用自己的声誉为公司背书。

#### Rule 5.5 — 可疑供应商 / 客户

```
数据: PDF 附注 "前五名客户" / "前五名供应商"
逻辑:
  - 前5大客户合计占比
  - 客户名称是否正常（如是否为关联方、壳公司）

PASS: 前5大客户占比 < 30%，名称正常
WARN: 前5大客户占比 > 50%，或客户为不透明实体
FAIL: 单一客户占比 > 50%
```

#### Rule 5.6 — 跨行业频繁收购

```
数据: PDF 董事会报告 "投资状况分析"
逻辑: 是否存在跨行业收购

PASS: 无跨行业收购或仅 1 次相关行业收购
WARN: 有跨行业收购
FAIL: 多次跨行业收购
```

#### Rule 5.7 — 商誉过大

```
数据: balance[latest].goodwill + total_hldr_eqy_exc_min_int
计算:
  goodwill_ratio = goodwill / total_hldr_eqy_exc_min_int × 100

PASS: ratio ≤ 20%
WARN: ratio > 20%
FAIL: ratio > 40%
```

商誉是收购溢价。太大 = 收购可能严重溢价，未来面临减值风险（商誉减值直接冲减利润）。

#### Rule 5.8 — 其他应收款异常

```
数据: balance[latest].oth_receiv + total_assets + PDF 年报
计算:
  ratio      = oth_receiv / total_assets × 100
  yoy_change = 其他应收款 YoY 变动%

PASS: ratio ≤ 3% 且 yoy_change ≤ 30%
WARN: ratio > 3% 或 yoy_change > 30%
FAIL: ratio > 5% 或 yoy_change > 50%

补充检查 (PDF): 年报全文中搜索 "非经营性占用"
  → 如该段落标注 "适用"，直接 FAIL
```

**背后逻辑**: 其他应收款是关联方资金占用的经典藏身之处。大额其他应收款 + 关联方往来 = 大股东占用上市公司资金。

#### Rule 5.9 — 监管处罚 / 立案调查

```
数据: PDF 年报全文
搜索: "立案" / "警示函" / "处罚决定" / "监管措施"
       / "行政处罚" / "纪律处分" / "通报批评" / "公开谴责"

A. 公司/控股股东/实控人被立案调查 → FAIL
B. 公司/董监高收到警示函或行政处罚 → WARN
C. 一般性合规整改、环保罚款 → PASS
D. 无匹配 → PASS
```

### Layer 6 — 行业特有风险（2 条规则）

#### Rule 6.1 — 农林渔牧行业

```
数据: stock_info.industry
风险关键词: 农业 / 林业 / 渔业 / 牧业 / 畜牧 / 养殖 / 种植 / 饲料 / 水产

PASS: 不属于以上行业
WARN: 行业包含以上关键词
```

**背后逻辑**: 生物资产难以审计（鱼塘里有多少鱼？山上有多少树？），历史上是造假重灾区（獐子岛、辉山乳业）。

#### Rule 6.2 — 研发资本化比例

```
数据: PDF 附注 "研发支出" / "开发支出"
搜索: "资本化"
计算:
  cap_ratio = 资本化金额 / (资本化金额 + 费用化金额) × 100

PASS: cap_ratio ≤ 30% 或全部费用化
WARN: cap_ratio > 30%
FAIL: cap_ratio > 50%
SKIP: 附注中无明确拆分
```

**背后逻辑**: 研发支出可以选择费用化（计入当期费用 → 减少利润）或资本化（计入资产 → 不影响当期利润）。资本化比例高 = 公司在美化利润。

## 5. 评分体系

### 5.1 逐条赋分表

| 层级 | 每条 WARN | 每条 FAIL | 说明 |
|------|----------|----------|------|
| Layer 1 - 利润表 | 2 分 | 5 分 | 利润表易操纵，权重中等 |
| Layer 2 - 现金流 | 3 分 | 6 分 | 现金流更难造假，权重较高 |
| Layer 3 - 资产负债表 | 2 分 | 5 分 | 权重中等 |
| Layer 4 - 交叉验证 | 3 分 | 7 分 | **权重最高**——三表交叉验证是核心 |
| Layer 5 - 非财务 | 1 分 | 3 分 | 辅助信号，权重较低 |
| Layer 6 - 行业 | 1 分 | 3 分 | 权重较低 |

总基础分 = Σ(各层 WARN 数 × WARN 分 + FAIL 数 × FAIL 分)

### 5.2 信号组合加分

某些信号组合同时出现意味着造假概率急剧上升，额外加分：

| 组合条件 | 加分 | 逻辑 |
|---------|------|------|
| Rule 3.2 FAIL（存货周转↓ + 毛利率↑） | **+10** | 虚构收入的黄金组合 |
| Rule 2.3 FAIL + Rule 4.1 FAIL（大存大贷 + CF/利润<1） | **+8** | 现金造假 + 利润无现金支撑 |
| Rule 1.2 FAIL + Rule 3.1 FAIL（毛利率↑应收↑ + 应收增速超收入）| **+6** | 虚增收入的完整信号链 |

### 5.3 风险等级映射

| 总分 | 风险等级 | 颜色 | 含义 |
|------|---------|------|------|
| 0 - 10 | 低风险 | 🟢 绿色 | 未发现明显异常 |
| 11 - 25 | 中风险 | 🟡 黄色 | 有若干关注信号，需进一步了解 |
| 26 - 45 | 高风险 | 🟠 橙色 | 多个异常信号，造假概率较高 |
| 46+ | 极高风险 | 🔴 红色 | 信号密集，强烈建议回避 |
| Layer 0 任一 FAIL | 直接排除 | ⚫ 黑色 | 基本可信度瓦解，一票否决 |

## 6. 数据库设计

### 6.1 核心表

```sql
-- 公司基本信息
CREATE TABLE stocks (
    id          SERIAL PRIMARY KEY,
    ts_code     VARCHAR(12) UNIQUE NOT NULL,  -- 如 600519.SH
    name        VARCHAR(100) NOT NULL,
    industry    VARCHAR(50),
    market      VARCHAR(10),
    list_date   DATE,
    is_active   BOOLEAN DEFAULT TRUE,
    created_at  TIMESTAMP DEFAULT NOW(),
    updated_at  TIMESTAMP DEFAULT NOW()
);

-- 年报审计信息
CREATE TABLE audit_records (
    id          SERIAL PRIMARY KEY,
    stock_id    INT REFERENCES stocks(id),
    end_date    DATE NOT NULL,        -- 报告期截止日
    ann_date    DATE NOT NULL,         -- 公告日期
    audit_result VARCHAR(50) NOT NULL, -- 审计意见
    audit_agency VARCHAR(100),         -- 会计师事务所
    audit_fees  DECIMAL(18,2),         -- 审计费用(万元)
    UNIQUE(stock_id, end_date)
);

-- 利润表 (年度)
CREATE TABLE income_statements (
    id              SERIAL PRIMARY KEY,
    stock_id        INT REFERENCES stocks(id),
    end_date        DATE NOT NULL,
    -- 核心科目 (单位: 元)
    revenue         DECIMAL(24,2),    -- 营业收入
    oper_cost       DECIMAL(24,2),    -- 营业成本
    sell_exp        DECIMAL(24,2),    -- 销售费用
    admin_exp       DECIMAL(24,2),    -- 管理费用
    rd_exp          DECIMAL(24,2),    -- 研发费用
    fin_exp         DECIMAL(24,2),    -- 财务费用
    assets_impair_loss  DECIMAL(24,2), -- 资产减值损失
    credit_impa_loss    DECIMAL(24,2), -- 信用减值损失
    oth_biz_income  DECIMAL(24,2),    -- 其他业务收入
    operate_profit  DECIMAL(24,2),    -- 营业利润
    n_income_attr_p DECIMAL(24,2),    -- 归母净利润
    basic_eps       DECIMAL(12,4),    -- 每股收益
    UNIQUE(stock_id, end_date)
);

-- 资产负债表 (年度)
CREATE TABLE balance_sheets (
    id              SERIAL PRIMARY KEY,
    stock_id        INT REFERENCES stocks(id),
    end_date        DATE NOT NULL,
    -- 核心科目 (单位: 元)
    money_cap           DECIMAL(24,2),  -- 货币资金
    accounts_receiv     DECIMAL(24,2),  -- 应收账款
    oth_receiv          DECIMAL(24,2),  -- 其他应收款
    inventories         DECIMAL(24,2),  -- 存货
    cip                 DECIMAL(24,2),  -- 在建工程
    fix_assets          DECIMAL(24,2),  -- 固定资产
    goodwill            DECIMAL(24,2),  -- 商誉
    lt_amort_deferred_exp DECIMAL(24,2),-- 长期待摊费用
    total_assets        DECIMAL(24,2),  -- 总资产
    st_borr             DECIMAL(24,2),  -- 短期借款
    lt_borr             DECIMAL(24,2),  -- 长期借款
    bond_payable        DECIMAL(24,2),  -- 应付债券
    acct_payable        DECIMAL(24,2),  -- 应付账款
    total_liab          DECIMAL(24,2),  -- 总负债
    total_hldr_eqy_exc_min_int DECIMAL(24,2), -- 归母净资产
    UNIQUE(stock_id, end_date)
);

-- 现金流量表 (年度)
CREATE TABLE cashflow_statements (
    id              SERIAL PRIMARY KEY,
    stock_id        INT REFERENCES stocks(id),
    end_date        DATE NOT NULL,
    -- 核心科目 (单位: 元)
    n_cashflow_act      DECIMAL(24,2), -- 经营CF
    n_cashflow_inv_act  DECIMAL(24,2), -- 投资CF
    n_cash_flows_fnc_act DECIMAL(24,2),-- 筹资CF
    c_recp_prov_sg_act  DECIMAL(24,2), -- 销售收现
    c_pay_acq_const_fiolta DECIMAL(24,2), -- Capex
    free_cashflow       DECIMAL(24,2), -- FCF
    UNIQUE(stock_id, end_date)
);

-- 财务指标 (年度)
CREATE TABLE financial_indicators (
    id              SERIAL PRIMARY KEY,
    stock_id        INT REFERENCES stocks(id),
    end_date        DATE NOT NULL,
    -- 核心指标
    roe                 DECIMAL(12,4), -- ROE (%)
    grossprofit_margin  DECIMAL(12,4), -- 毛利率 (%)
    netprofit_margin    DECIMAL(12,4), -- 净利率 (%)
    inv_turn            DECIMAL(12,4), -- 存货周转率
    ar_turn             DECIMAL(12,4), -- 应收周转率
    assets_turn         DECIMAL(12,4), -- 总资产周转率
    debt_to_assets      DECIMAL(12,4), -- 资产负债率 (%)
    revenue_yoy         DECIMAL(12,4), -- 营收YoY (%)
    netprofit_yoy       DECIMAL(12,4), -- 净利YoY (%)
    fcff                DECIMAL(24,2), -- 企业自由现金流
    interestdebt        DECIMAL(24,2), -- 有息负债
    UNIQUE(stock_id, end_date)
);

-- 股东数据
CREATE TABLE top_holders (
    id          SERIAL PRIMARY KEY,
    stock_id    INT REFERENCES stocks(id),
    end_date    DATE NOT NULL,
    holder_name VARCHAR(200) NOT NULL,
    hold_amount DECIMAL(18,2),        -- 持股数量(股)
    hold_ratio  DECIMAL(8,4),         -- 持股比例(%)
    rank        INT                   -- 排名
);

-- 同行对比数据
CREATE TABLE peer_comparisons (
    id              SERIAL PRIMARY KEY,
    stock_id        INT REFERENCES stocks(id),
    peer_stock_id   INT REFERENCES stocks(id),
    end_date        DATE NOT NULL,
    grossprofit_margin  DECIMAL(12,4),
    netprofit_margin    DECIMAL(12,4),
    roe                 DECIMAL(12,4),
    debt_to_assets      DECIMAL(12,4),
    inv_turn            DECIMAL(12,4),
    ar_turn             DECIMAL(12,4)
);

-- 排雷分析记录
CREATE TABLE finscan_reports (
    id          SERIAL PRIMARY KEY,
    stock_id    INT REFERENCES stocks(id),
    report_year INT NOT NULL,          -- 分析年度 (如 2024)
    total_score DECIMAL(8,2),          -- 总分
    risk_level  VARCHAR(20),           -- LOW / MEDIUM / HIGH / VERY_HIGH / REJECT
    n_pass      INT DEFAULT 0,
    n_warn      INT DEFAULT 0,
    n_fail      INT DEFAULT 0,
    n_skip      INT DEFAULT 0,
    combo_bonus INT DEFAULT 0,         -- 组合加分
    data_source VARCHAR(50),           -- tushare / akshare
    pdf_used    BOOLEAN DEFAULT FALSE,
    raw_json    JSONB,                 -- 完整原始数据
    created_at  TIMESTAMP DEFAULT NOW()
);

-- 逐条规则结果
CREATE TABLE rule_results (
    id          SERIAL PRIMARY KEY,
    report_id   INT REFERENCES finscan_reports(id),
    layer       INT NOT NULL,          -- 0-6
    rule_code   VARCHAR(10) NOT NULL,  -- 如 "1.2", "3.2", "4.1"
    rule_name   VARCHAR(100),
    verdict     VARCHAR(10) NOT NULL,  -- PASS / WARN / FAIL / SKIP
    score_added DECIMAL(8,2) DEFAULT 0,
    detail      TEXT,                  -- 判定详情 (含实际数值)
    calculation JSONB                  -- 中间计算过程
);
```

```
  → { report: { score, risk_level, rules: [...], findings: [...] } }

GET  /api/v1/finscan/reports/{report_id}/rules/{rule_code}
  → { rule: { data, calculation, verdict, detail } }

# 多股对比
POST /api/v1/finscan/compare
  Body: { stock_codes: ["600519", "000858", "002304"], year: 2024 }
  → { comparison: [{stock, score, level, fail_rules: [...]}] }

# 自选股列表
GET    /api/v1/watchlist
POST   /api/v1/watchlist
DELETE /api/v1/watchlist/{stock_id}

# 历史扫描记录
GET  /api/v1/history?stock_code=600519
  → { reports: [{year, score, risk_level, created_at}] }
```

### 7.2 分析任务流程

```
POST /analyze → 创建 task → 返回 task_id
         │
         ├── Step 1: 数据采集 (10%)
         │      ├── Tushare/akshare 拉取结构化数据
         │      └── 巨潮资讯网下载年报 PDF
         │
         ├── Step 2: PDF 解析 (20%)
         │      └── pdfplumber 提取文本 → 正则提取关键字段
         │
         ├── Step 3: 规则评估 (60%)
         │      └── 逐条运行 30 个规则，记录 PASS/WARN/FAIL/SKIP
         │
         ├── Step 4: 评分计算 (10%)
         │      └── 基础分 + 组合加分 → 风险等级映射
         │
         └── Step 5: 报告生成 (100%)
                └── 写入数据库 → task 状态设为 done
```

## 10. 关键注意事项

### 10.1 数据精度

- Tushare/akshare 返回的数值单位可能是**元**，展示时需除以 10^4（万元）或 10^8（亿元）
- 毛利率、ROE 等百分比指标注意是百分数（如 33.67 表示 33.67%）还是小数
- 减值损失可能是负数（表示损失），计算 YoY 时注意绝对值处理

### 10.2 同行对比的坑

- 同行样本可能很少（部分行业只有 3-5 家公司），此时用中位数不够稳健
- 同行数据可能和标的公司不同期，需确保取同一 `end_date`
- 建议同行样本 < 3 家时自动标注"样本不足"但仍输出对比结果

### 10.3 PDF 解析的难点

- 年报 PDF 格式不统一，不同公司的附注命名和结构差异大
- 运费可能叫"运输费"/"物流费"/"装卸费"，需要维护关键词库
- 研发资本化拆分可能在一段文字中而非表格中
- 建议对 PDF 解析失败率设定合理的容忍度（约 20-30% 的 PDF 依赖规则可能 SKIP）

### 10.4 规则阈值调优

- 当前阈值基于书中的经验值，上线后应根据实际数据持续校准
- 建议在管理后台暴露阈值参数，便于后续调优
- 不同行业的"正常"范围不同（如零售业毛利率低但周转快，药企毛利率高），后续可引入分行业阈值

### 10.5 与 LLM 的关系

- **本 Web 系统中不需要 LLM**：所有 30 条规则都是确定性计算
- LLM 仅用于原始 Claude Code Skill 中做自然语言理解和报告生成
- Web 版中规则引擎是纯 Python 函数，报告是模板渲染
